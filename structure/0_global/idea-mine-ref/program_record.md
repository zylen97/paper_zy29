## 0. 一句话目标

借鉴徐进两篇"工程装备知识跨项目迁移"的内核，建立一个**研究纲领（非单篇）**：围绕同一内核做 **一软一硬两篇论文**，硬篇投 Automation in Construction / AEI，软篇投 IEEE TEM / IJPM / JME。

**作者约束（硬约束，决定一切取舍）**
- 偏**建设工程管理 / 知识管理**背景，非算法/CV 出身。
- **没有任何私有数据集**（无盾构、无传感器数据）→ 选题必须有公开数据。
- 要"硬"（拒绝纯管理统计预测那种软任务）。

## 1. 源头：徐进两篇拆解（papers/ 目录下两 PDF）

- **2024 MTFF**：特征差异增强的多源迁移；**2026 TBMKF**：时序分块 + 多知识融合。
- 两篇 = 同数据(8 个盾构项目) + 同任务(盾构姿态偏差 HDSH/VDSH 预测) + 同内核，只换方法学组件。都发在《系统工程理论与实践》(管理刊)。
- **本质打法**：硬任务(盾构控制) + 包"工程项目知识管理/智能建造"软壳，发管理刊。

**"迁移"到底迁什么（核心概念，已厘清）**
- 迁的不是数据/文档，是**机器学到的"规律"**（输入→输出的映射，固化在模型参数里）。
- 徐进迁的是"传感器读数 → 盾构姿态偏差"这条规律。
- 动机：新项目数据太少，自己学不出可靠规律 → **借老项目学好的规律**。
- 相似度测度：拿老项目模型去预测新项目数据，看预测准不准(R²)反推相似度，选 Top-E 个最像的源。
- 防"负迁移"：迁错了(软土经验套岩石)会帮倒忙，所以要挑相似的、防借错。
- 多源融合：既要共识(共性知识)，也要每个源的绝活(域特殊知识)。

## 2. ✅ 统一内核（这把伞撑起软硬两篇）

> **Similarity-gated, negative-transfer-aware, interpretable cross-unit knowledge transfer**
> 相似度门控 + 负迁移感知 + 可解释的**跨单元**知识迁移

- 硬任务上 → 一个 computational framework（迁移算法）→ AC/AEI
- 软任务上 → 一个管理命题（什么驱动有效迁移、负迁移的组织机理）→ TEM/IJPM
- 两者互为表里：方法验证理论，理论指导方法。

---

## 3. ✅ 关键战略教训（最重要，避免重蹈覆辙）

**新颖性不住在"迁移机制"里。** 负迁移、相似度选源、可解释——这些在所有应用域(建筑能耗、轴承故障、热舒适…)都已饱和。徐进能发，**不是方法新**（他用标准 MMD/域适应），**而是"盾构施工知识"是当时还没人用迁移做的新鲜硬应用域 + 管理壳**。

> **护城河 = 新鲜的硬应用域 × 管理 framing，不是算法本身。**

**"硬"的三条判据**（用来筛任务）：
1. 作用在**物理对象/施工过程**上（设备、结构、地层），不是作用在账本上；
2. **预测→动作闭环**：预测能直接触发工程动作（纠偏、调参、控制、养护决策），不是写进报告；
3. **高频 + 明确工程后果**。

## 5. 🟡 硬篇：三个候选（仍在讨论，**未拍板**）

> 三个的共同点：迁移楔子都还新鲜(未被占)。差别在"数据可得性 / 作者能否驾驭 / 离工程管理近不近"。

### 候选 1：基础设施劣化跨资产迁移
- **大白话**：路、桥建好后会慢慢变坏，养护部门定期给每座桥每段路打"健康分"(美国 NBI 给桥构件打 1–9 分)。用老桥老路几十年的评分变化，预测新桥新路未来会坏到什么程度 → 安排维修、分预算。新桥没历史记录 → 从老桥借"特征→劣化速度"的规律。
- **硬**：养护决策闭环。 **数据**：✅✅最强（LTPP 路面、NBI/NBE 桥梁构件评级，公开、几十万条、跨几十年面板）。 **管理壳**：资产组合管理(很浓)。
- **作者顾虑**：偏"运维"不是"施工建造"，离建设项目管理稍远；与软篇有"资产 vs 项目"接缝。

### 候选 2：活动识别 → 生产率跨项目迁移
- **大白话**：工地装摄像头/给设备工人戴传感器，电脑自动认"挖机在挖土/回转/装车""工人在搬运/绑钢筋"，累加算出生产率/进度。A 工地训练的识别器搬到 B 工地(角度光照设备不同)→ 要迁移适配。
- **硬**：作用在工地物理过程，识别→生产率/进度→调度。 **数据**：✅有公开视频/IMU 活动识别集。 **管理壳**：生产率/进度管理(正中本行)，属"智能建造"，离工程管理近。 **与软篇耦合最紧(都是"项目"层)**。
- **代价**：硬任务本质是**计算机视觉**，作者可能驾驭/审稿吃力；直接"生产率"标签弱，多半停在动作识别、生产率成解释层。
- 作者目前**对此方向有好感**（觉得贴智能建造/工程管理），但**明确表示尚未选定，要三个一起再讨论**。

### 候选 3：深基坑变形预测跨项目迁移
- **大白话**：盖楼前挖的地下大坑，开挖时坑壁会变形、可能塌、危及邻楼。埋传感器实时测，预测接下来变形多少 → 及时加支撑。新基坑数据少 → 从老基坑借"开挖→变形"规律。跟徐进盾构是孪生兄弟。
- **硬**：✅最硬(支护/开挖闭环、关安全)。 **新鲜度**：✅✅最新鲜(楔子全空)。 **数据**：❌弱(监测数据多私有，无公开集) = 和盾构同陷阱。
- 除非能从合作单位搞到一批多项目基坑监测数据，否则做不了。

**三者速览**

| | 通俗 | 数据 | 作者能否驾驭 | 离工程管理 |
|---|---|---|---|---|
| 1 劣化预测 | 看老桥老路怎么坏预测新的 | ✅公开海量 | ✅不碰CV | 偏运维，较远 |
| 2 活动识别 | 摄像头认工地动作算生产率 | ✅有公开集 | ⚠️要会CV | 近(智能建造) |
| 3 基坑变形 | 看老基坑预测新基坑 | ❌无公开数据 | ✅但没数据白搭 | 近 |

**主 agent 的推荐（仅供参考，非结论）**：按"无私有数据"硬约束，候选 3 先排除(数据死穴)；候选 1 数据与管理壳最强但偏运维；候选 2 最贴工程管理但偏 CV。**最终由作者拍板，当前仍在讨论。**

## 6. 🟡 软篇：方向基本锁定（细节待展开）

**"项目组织的知识负迁移治理"** → 投 IJPM / JME / IEEE TEM。

- **问题**：项目型组织(设计院/施工企业)都说"复用老项目经验"，但**不是所有老经验都该复用，用错会害新项目(负迁移)**。何时复用会帮倒忙？新项目该向哪些老项目学/不学？组织该建什么机制管？
- **空白证据**：OpenAlex 查"negative transfer + project management"命中=**1**(还不相关)，管理侧近乎真空。管理界一直默认"知识共享=越多越好"，没人做风险侧。
- **贡献**：提出"**可解释地挑该向谁学 = 一种新的知识治理机制**"，把"该向谁学"从隐性直觉变成可制度化决策。
- **避坑**：纯"文化/机制影响知识转移"已饱和(Prencipe&Tell 2001、Wiewiora 2013、Ren 2018、Zhang 2020 等已占)；"cross-project transfer"一词被软件工程缺陷预测 CPDP 占用，须用 project-based organization 框定。
- 可把"项目相似度度量"(建设工程侧很薄)作为软硬共用的"总开关"概念。

## 7. ✅ 一软一硬如何咬合成一套（不是两摊）

- **硬篇**：在具体工程对象(桥/工地/基坑)上，真做出"挑对源、防负迁移"的算法，拿硬数据证明"挑对源后绩效提升多少"。
- **软篇**：把这套做法上升为管理道理——"知识复用非越多越好，存在负迁移风险，可解释源选择=新治理机制"，并**引用硬篇实证当证据**。
- **软篇讲道理，硬篇给证据，共享"源选择+负迁移"内核** → 连贯系列。
- 连贯性两座桥：①方法桥(共用迁移引擎)②证据桥(硬篇产实证规律→软篇理论化)。**硬篇须先写**(软篇要它的证据)。
